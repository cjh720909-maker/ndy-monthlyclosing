# NeonDB 프로젝트별 테이블 격리 운영 가이드라인

본 문서는 여러 프로젝트가 하나의 NeonDB 인스턴스를 공유할 때 발생하는 데이터 혼선을 방지하고, 각 프로젝트의 독립성을 보장하기 위한 표준 운영 절차를 정의합니다.

## 🏗️ 운영 원칙: [중앙 관리 & 개별 실행]

### 1. 역할 분담 (Role Separation)

#### **[중앙 관리자 (안팀장)]**
- **NeonDB 환경 관리**: 공용 DB 내에 `prj_` 접두사를 가진 전용 방(Schema)을 생성합니다. (예: `prj_monthly_closing`)
- **방(Schema) 할당**: 신규 프로젝트 요청 시 전용 스키마를 생성하고 통보합니다.

#### **[각 프로젝트 담당자 (개별 프로젝트)]**
- **테이블 네이밍 (Header)**: 각 프로젝트는 고유한 약어 접두사를 사용합니다.
    - 예: `ndy-monthly-closing` -> **`NMC_`** (`N`DY + `M`onthly + `C`losing)
- **이사(Migration) 실행**: 할당된 방으로 기존 데이터를 옮기는 작업은 **해당 프로젝트 폴더 내**에서 직접 실행합니다.
- **설정 변경**: `.env` 파일을 수정하여 발급받은 전용 방으로 연결합니다.
- **검증**: 자신의 서비스가 새로운 방에서 정상 작동하는지 확인합니다.

---

## 📩 프로젝트 적용 요청 사항 (Requirement)

모든 프로젝트는 아래 4가지 핵심 사항을 반드시 준수해야 합니다.

### 1. 네이밍 규칙 준수
- 전용 스키마 명칭은 반드시 `prj_[프로젝트명]` 형식을 사용합니다.
- 테이블명은 반드시 프로젝트 고유 접두사(Header)로 시작해야 합니다.

### 2. .env 연결 설정 업데이트
- `DATABASE_URL` 끝에 `?schema=prj_[프로젝트명]` 파라미터를 반드시 추가합니다.

### 3. 안전한 데이터 이관
- `public` 스키마에서 기존 데이터를 가져올 때는 제공된 안전 이관 스크립트(`migrate_to_schema.js`)를 활용하여 데이터 유실을 방지합니다.

### 4. 절대 금지 조항
- `npx prisma db push --force-reset` 등 DB를 초기화하는 명령어를 절대 사용하지 않습니다.
- 타 프로젝트의 전용 방(Schema)을 무단으로 참조하거나 수정하지 않습니다.
